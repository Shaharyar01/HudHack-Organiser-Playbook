# 🏆 Devpost & Judging Playbook

Managing submissions and judging at a hackathon can easily become chaotic if you aren't organized. For HudHack 2026, we used **Devpost** to handle project submissions and a **Roaming Judge** format to evaluate teams on the floor.

Here is the complete guide on how to run this process for future events.

## 🌐 Why We Use Devpost

[Devpost](https://devpost.com/) is the global industry standard for hackathons. We highly recommend sticking with it for a few key reasons:
1. **The Project Gallery:** It automatically creates a beautiful, permanent gallery of every project built at your event. You can view our 2026 gallery here: https://hudhack2026.devpost.com/.
2. **Student Portfolios:** When a student submits a project on Devpost, it gets added to their public developer portfolio, which is fantastic for their CVs.
3. **Submission Management:** It forces teams to write a description, list the tech stack they used, and provide a link to their GitHub repository all in one standardized format.

### 🛠️ How to Set Up Devpost
Setting up the hackathon on Devpost is free for university/student events. 
1. Create a Manager account on Devpost and click "Post a Hackathon".
2. Fill out the public details (Date, Theme, Prizes).
3. **Set strict deadlines:** Ensure the submission deadline is set to the exact time hacking stops (e.g., 16:45). Devpost will automatically lock submissions at this time, preventing teams from coding after the buzzer.
4. **Custom Questions:** Make sure to require a link to a public GitHub repository so judges can actually look at the code!

---

## ⚖️ The Judging Process

At 16:45, all hacking stopped. Instead of making teams present on a stage (which takes hours), we used a "Science Fair" roaming judge format.

1. We printed out physical Judging Sheets (included in this folder).
2. Judges walked around the room, going table to table.
3. Each team had strictly **3-5 minutes** to pitch their idea and show a live demo.
4. Judges filled out the scorecards and then convened in a private room to tally the scores and determine the winners before the Closing Ceremony.

---

## 📋 The 2026 Judging Criteria

We used a standardized 30-point rubric to ensure all teams were graded fairly. The maximum score a project could receive was 30 points. 

Judges scored each category on a scale from 1 to 5, where 1 represents "Weak / Limited", 3 represents "Good / Meets Expectations", and 5 represents "Excellent / Outstanding".

Here are the 6 categories we assessed:
* **Theme Alignment:** Connection to the theme. (e.g., Did it clearly relate to "Break The Loop"?)
* **Innovation & Creativity:** Originality of the idea.
* **Technical Achievement:** Difficulty & quality of what was built. 
* **Functionality:** Does it work?. (e.g., Is it incomplete, or are the core features fully working?)
* **Impact & Usefulness:** Real-world value and strong potential.
* **Presentation & Demo:** Clarity of the explanation and an engaging demo.

### 🚨 The Tie-Breaker
In the event of a tie between two top projects, judges used a single tie-breaker question: 
* **"Which project would you most want to see continued after HudHack?"**.

---

## 💡 Advice for 2027

* **Scale Up with Excel:** We used printed paper scorecards this year, which worked perfectly for ~10 projects. However, **if the hackathon is bigger next year, we highly recommend using a shared Excel or Google Sheet** for the judges to input their scores on their phones/laptops. Tallying 30+ paper scorecards by hand will delay the closing ceremony!
